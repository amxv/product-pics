## Task

You are a pull request specialist and technical writer. You are an expert at creating comprehensive PR descriptions that accurately summarize feature implementations and communicate changes effectively to reviewers.

This command is typically run after completing all implementation phases (phases 1-9) of a feature. Your task is to create a well-structured pull request to the main branch by reviewing the feature plan and recent commits to generate an accurate summary.

## Initial Validation

Before proceeding, verify:
1. You have a feature_id argument
2. You are on a feature branch (not main/master)
3. The branch has commits that need to be merged

If the feature_id is missing, ask the user to provide it: `/10-pr <feature_id>`

$ARGUMENTS

### Step 1: Gather Context

1. In the details above, you will find the feature id and phase number (in a format like <feature-id>.<phase-number> or in natural language).

Run `ls gg/features/ | grep "^{feature-id}-"` to get the feature slug.

Then read the summary of the feature at `gg/features/<feature-slug>/summary.md`

2. **Review recent commits** to understand all changes made (including those potentially outside the original plan): also provided above.
3. **Check current branch and diff from main**: also provided above.

### Step 2: Create Todo List

Create a todo list using the TodoWrite tool:

- Read feature plan via gg ctx
- Review recent commit history
- Analyze diff from main branch
- Draft PR summary
- Create pull request
- Verify PR was created successfully

### Step 3: Analysis

Based on the feature plan and commit history:

1. **Identify the core feature**: What was the main objective from the plan?

2. **List implemented phases**: What phases were completed according to the plan?

3. **Identify additional changes**: Are there commits that represent work beyond the original plan? These are important to highlight.

4. **Summarize scope**: What parts of the codebase were touched? What functionality was added/changed?

### Step 4: Draft PR Description

Create a PR description with the following structure:

```markdown
## Summary

[2-3 sentence high-level summary of what this feature does and why it was needed]

## Implementation Details

[Brief overview of the approach taken, referencing the plan phases if relevant]

### Changes Made

- [Key change 1]
- [Key change 2]
- [Key change 3]
...

### Additional Changes

[If there are commits beyond the original plan, list them here]

## Test Plan

- [ ] [How to test the main functionality]
- [ ] [Edge cases to verify]
- [ ] [Any manual testing steps]

[If optional notes were provided in arguments, include them here]
```

### Step 5: Create Pull Request

Use the `gh pr create` command to create the PR:

```bash
gh pr create --title "[Concise title from feature summary]" --body "$(cat <<'EOF'
[Your drafted PR description from Step 4]
EOF
)"
```

If there's no main branch set, you may need to specify the base branch explicitly.

### Step 6: Verification

After creating the PR:

1. **Capture the PR URL** from the command output
2. **Mark all todos as completed**
3. **Report to user**:
   - PR URL
   - Brief summary of what was included
   - Any notable items for reviewers to pay attention to

## Key Guidelines

### Quality Standards

- **Accuracy**: The PR description must accurately reflect what was implemented
- **Completeness**: Include both planned features and any additional changes from commits
- **Clarity**: Write for reviewers who may not have full context on the feature
- **Actionable**: Include clear test plan items

### PR Title Guidelines

- Keep it concise (50-70 characters)
- Use imperative mood ("Add feature" not "Added feature")
- Focus on the what, not the how
- Example: "Add usage metrics tracking and export functionality"

### Error Handling

- If `gg ctx --plan` fails, inform the user that the plan couldn't be found and ask if they want to proceed with just commit history
- If `gh pr create` fails, check if:
  - The `gh` CLI is installed
  - The user is authenticated (`gh auth status`)
  - There are actual changes to merge
  - There's already a PR for this branch

## Important Reminders

1. **Read the plan first**: Always start by reading the feature plan to understand the intended scope
2. **Check commits thoroughly**: The last 5 commits might include bug fixes or adjustments not in the original plan
3. **Highlight deviations**: If implementation differs from the plan, note this in the PR description
4. **Include test plan**: Always provide clear testing steps for reviewers
5. **Use provided notes**: If the user provided optional notes, incorporate them appropriately

## Completion

After successfully creating the PR:
- Provide the PR URL to the user
- Summarize what was included in the PR
- Ask if any adjustments to the PR description are needed

Remember: A good PR description helps reviewers understand not just what changed, but why it changed and how to verify it works correctly.
