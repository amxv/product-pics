## Phase 1: Authentication, Database, and Foundation

### Overview

This phase establishes the foundation for the entire application by setting up authentication, database schema, shared types, and core infrastructure. After this phase, users can sign up, sign in, and the database is ready to store batches and images.

### Tasks

- [ ] 1. Install Dependencies
  - [ ] 1.1 Install production dependencies: `bun add better-auth drizzle-orm postgres @aws-sdk/client-s3 @aws-sdk/s3-request-presigner zod nanoid`
  - [ ] 1.2 Install development dependencies: `bun add -D drizzle-kit @better-auth/cli`

- [ ] 2. Configure Environment Variables
  - [ ] 2.1 Create `.env.example` with all required variables (DATABASE_URL, BETTER_AUTH_SECRET, BETTER_AUTH_URL, R2 credentials, RUNPOD_API_KEY)
  - [ ] 2.2 Add instructions for copying to `.env` and filling in actual values

- [ ] 3. Create Database Schema
  - [ ] 3.1 Define pgEnums in `db/schema.ts` (demographicEnum, batchStatusEnum, uploadStatusEnum, generationStatusEnum, runpodJobStatusEnum)
  - [ ] 3.2 Define better-auth tables (userTable with username extension, sessionTable, accountTable, verificationTable)
  - [ ] 3.3 Define application tables (batchTable, uploadedImageTable, generatedImageTable, runpodJobTable)
  - [ ] 3.4 Add foreign key relationships with `onDelete: 'cascade'` for child records
  - [ ] 3.5 Add indexes on frequently queried columns (userId, batchId, status fields, createdAt)

- [ ] 4. Configure Drizzle
  - [ ] 4.1 Create `drizzle.config.ts` with schema path, migrations output, PostgreSQL dialect, and database credentials

- [ ] 5. Create Drizzle Client
  - [ ] 5.1 Create `db/index.ts` with Drizzle client instance using `drizzle()` from `drizzle-orm/postgres-js`
  - [ ] 5.2 Export db instance and all schema tables for use throughout application

- [ ] 6. Create Shared Types
  - [ ] 6.1 Define all enums and type aliases in `lib/types.ts` (DEMOGRAPHICS, BATCH_STATUSES, BACKGROUNDS, SUPPORTED_FORMATS, etc.)
  - [ ] 6.2 Define database types using InferSelectModel and InferInsertModel from Drizzle
  - [ ] 6.3 Define API request/response types (PresignedUrlRequest, CreateBatchRequest, BatchWithImages, BatchProgress)
  - [ ] 6.4 Define RunPod API types (RunpodSubmitRequest, RunpodStatusResponse)
  - [ ] 6.5 Create Zod validation schemas (ageRangeSchema, createBatchSchema, presignedUrlRequestSchema)

- [ ] 7. Configure better-auth
  - [ ] 7.1 Create server-side auth instance in `lib/auth.ts` with Drizzle adapter, emailAndPassword, username plugin, and session expiration
  - [ ] 7.2 Create client-side auth utilities in `lib/auth-client.ts` using `createAuthClient()` from better-auth/react

- [ ] 8. Create Auth API Route
  - [ ] 8.1 Create `app/api/auth/[...all]/route.ts` with GET and POST handlers that call `auth.handler()`

- [ ] 9. Create R2 Client
  - [ ] 9.1 Create S3Client instance in `lib/r2.ts` configured for Cloudflare R2 (region: 'auto', endpoint, forcePathStyle: true)
  - [ ] 9.2 Export helper functions: `generateUploadKey()`, `generateGeneratedKey()`, `getPublicUrl()`

- [ ] 10. Create Auth Pages
  - [ ] 10.1 Create auth layout in `app/(auth)/layout.tsx` with centered container (no navbar)
  - [ ] 10.2 Create sign-in page in `app/(auth)/sign-in/page.tsx` with username/password form using shadcn/ui components
  - [ ] 10.3 Create sign-up page in `app/(auth)/sign-up/page.tsx` with name/username/email/password form using shadcn/ui components

- [ ] 11. Create Protected App Layout
  - [ ] 11.1 Create app layout in `app/(app)/layout.tsx` with session check, navbar with user info and sign-out button
  - [ ] 11.2 Create home page in `app/(app)/page.tsx` that redirects to `/batches`

- [ ] 12. Create Middleware
  - [ ] 12.1 Create `middleware.ts` for optimistic route protection (check session cookie, redirect to /sign-in if missing)

- [ ] 13. Run Migrations
  - [ ] 13.1 Generate migration files: `bunx drizzle-kit generate`
  - [ ] 13.2 Run migrations against database: `bunx drizzle-kit migrate`

- [ ] 14. Configure R2 Lifecycle Policies
  - [ ] 14.1 Set up 30-day deletion rule for `uploads/` prefix in R2 bucket settings
  - [ ] 14.2 Set up 30-day deletion rule for `generated/` prefix in R2 bucket settings

- [ ] 15. Configure R2 CORS
  - [ ] 15.1 Configure CORS rules for R2 bucket (allow PUT/GET methods, Content-Type header, app domain origins)

### Manual Tasks

The following tasks must be completed manually before or during Phase 1 implementation:

1. **Set up PostgreSQL Database**
   - Create a PostgreSQL database locally or via cloud provider (Neon, Supabase, Railway)
   - Database name: `product-pics`
   - Update `DATABASE_URL` in `.env` with connection string

2. **Generate better-auth Secret**
   - Run: `openssl rand -base64 32`
   - Copy output to `BETTER_AUTH_SECRET` in `.env`

3. **Set up Cloudflare R2**
   - Create Cloudflare account (if needed)
   - Navigate to R2 and create bucket: `product-pics`
   - Generate R2 API token with read/write permissions
   - Copy Account ID, Access Key ID, and Secret Access Key to `.env`
   - Configure CORS rules (Task 15.1)
   - Configure lifecycle policies for 30-day deletion (Tasks 14.1 and 14.2)

4. **Set up RunPod Account**
   - Sign up at runpod.io
   - Navigate to API Keys and generate new key
   - Copy API key to `RUNPOD_API_KEY` in `.env`
   - Verify Seedream 4 Edit model is available at endpoint: `https://api.runpod.ai/v2/seedream-v4-edit`

5. **Test Authentication Flow**
   - Start dev server: `bun dev`
   - Navigate to `/sign-up`
   - Create test account
   - Verify redirect to `/batches` (will be empty, but should not error)
   - Sign out and sign back in
   - Verify session persists across page refreshes

6. **Verify Database Schema**
   - Run: `bunx drizzle-kit studio`
   - Open Drizzle Studio in browser to inspect database tables
   - Verify schema matches design from phase plan
