# 001-ai-product-pics - Phase 4: Download and Edge Cases

## Phase Overview

This phase implements the download functionality and handles all edge cases. Users can download completed batches as zip files containing all generated images and a summary report. The system handles duplicate filenames, partial results, errors, and other edge cases gracefully. After this phase, the feature is fully functional and production-ready.

**Dependencies**: This phase depends on Phase 3 being complete (images generated and stored in R2).

**References**:
- See `gg/features/001-ai-product-pics/summary.md` for the complete feature overview and implementation approach.
- See Phase 1 plan for Database Schema and Shared Type Definitions.
- See Research document for zip generation strategies (on-demand, not pre-generated).

#### Patterns, Conventions, and Design Decisions to Reuse

**Download Flow:**
1. User clicks "Download Results" button on batch details page
2. Client initiates download from `/api/download/[batchId]`
3. Server:
   - Validates session and batch ownership
   - Verifies batch status is "completed" or "partial"
   - Fetches all generated images with status="completed"
   - Generates summary report (text file)
   - Creates zip file with streaming (archiver library)
   - Streams zip to client (progressive download)
4. Client receives zip file: `batch-{batchId}-{timestamp}.zip`
5. Zip contains:
   - All generated images: `{originalFilename}_generated.png`
   - Summary report: `SUMMARY.txt`

**Zip Generation Pattern (On-Demand):**
- Do NOT pre-generate and store zips (wastes storage)
- Generate on-demand when requested
- Stream zip to client (start download immediately, no buffering)
- Use `archiver` library for efficient streaming
- Handle large batches (100 images) without memory issues

**Duplicate Filename Handling:**
- Track used filenames in Set during zip generation
- If duplicate detected, append `_1`, `_2`, etc.
- Example: `product.jpg` → `product_generated.png`, `product_generated_1.png`

**Summary Report Format:**
```
AI PRODUCT PHOTO GENERATION - SUMMARY REPORT
============================================

Batch ID: abc123
Generated: 2025-11-13 14:30:00
Demographic: Boy
Age Range: 8-10

STATISTICS
----------
Total Images: 20
Successfully Generated: 18
Failed: 2

SUCCESSFUL IMAGES
-----------------
1. product-001.jpg → product-001_generated.png (Background: playground)
2. product-002.jpg → product-002_generated.png (Background: beach)
...

FAILED IMAGES
-------------
1. product-015.jpg - Error: RunPod API timeout
2. product-018.jpg - Error: Safety checker flagged content

NOTES
-----
- All generated images are in PNG format at 1024×1024 resolution
- Images are retained for 30 days from generation date
- For support, contact: support@example.com
```

#### Key Constraints to work within

- **Streaming required for large zips** - Cannot buffer 100MB+ in memory
- **No pre-generated zips** - Generate on-demand to save storage costs
- **Must handle duplicate filenames** - Users may upload files with same name
- **Summary report must be accurate** - Match exactly what's in the zip

## Changes Required

### 1. Create Zip Generation Utility

**File**: `lib/zip.ts`

**Purpose**: Generate zip files with streaming

**Functions:**

**1. `createBatchZip(batchId: string, userId: string): Promise<{ stream: NodeJS.ReadableStream, filename: string }>`**

Generate zip file for batch:
- Input: batchId, userId (for verification)
- Steps:
  1. Fetch batch from database
  2. Verify batch status is "completed" or "partial"
  3. Fetch all generated_image records with status="completed"
  4. Fetch corresponding uploaded_image records for original filenames
  5. Create archiver instance: `archiver('zip', { zlib: { level: 6 } })`
  6. Track used filenames in Set for duplicate detection
  7. For each generated image:
     - Get original filename from uploaded_image
     - Generate output filename: `{originalFilename}_generated.png`
     - If duplicate, append `_1`, `_2`, etc.
     - Fetch image from R2 (GetObjectCommand)
     - Append to zip: `archive.append(imageBuffer, { name: outputFilename })`
  8. Generate summary report text
  9. Append summary to zip: `archive.append(summaryText, { name: 'SUMMARY.txt' })`
  10. Finalize archive: `await archive.finalize()`
  11. Return stream and filename
- Output: `{ stream: Readable, filename: string }`
- Throws error if batch not found or no images to download

**2. `generateSummaryReport(batch: Batch, generatedImages: GeneratedImage[], uploadedImages: UploadedImage[]): string`**

Generate summary report text:
- Input: Batch data, generated images, uploaded images
- Format: See "Summary Report Format" above
- Include:
  - Batch metadata (ID, demographic, age range, timestamp)
  - Statistics (total, completed, failed)
  - List of successful images with backgrounds
  - List of failed images with error messages
  - Notes section
- Output: Formatted text string

**3. `handleDuplicateFilename(filename: string, usedFilenames: Set<string>): string`**

Handle duplicate filenames:
- Input: Desired filename, set of used filenames
- Logic:
  - If filename not in set, return as-is and add to set
  - If duplicate, try `{name}_1.{ext}`, `{name}_2.{ext}`, etc.
  - Find first unused variant and return
- Output: Unique filename
- Side effect: Adds filename to usedFilenames set

### 2. Create Download API Route

**File**: `app/api/download/[batchId]/route.ts`

**Method**: GET

**Purpose**: Stream zip file download to client

**Logic**:
1. Validate session (must be authenticated)
2. Get batchId from params
3. Fetch batch from database
4. Verify user owns batch
5. Verify batch status is "completed" or "partial"
6. Call `createBatchZip(batchId, userId)` to generate zip stream
7. Set response headers:
   - `Content-Type: application/zip`
   - `Content-Disposition: attachment; filename="{filename}"`
   - `Cache-Control: no-cache`
8. Stream zip to response using NextResponse with stream
9. Return streaming response

**Response**: Binary zip file stream

**Errors**:
- 401 if not authenticated
- 403 if user doesn't own batch
- 400 if batch status is not "completed" or "partial"
- 404 if batch not found or no images to download

**Filename Format**: `batch-{batchId}-{timestamp}.zip`
- Example: `batch-abc123-20251113143000.zip`

### 7. Add Edge Case Handling

**File**: `lib/zip.ts`

**Additional Logic**:

**Handle Empty Batches:**
```typescript
if (generatedImages.length === 0) {
  throw new Error('No images available for download');
}
```

**Handle R2 Fetch Errors:**
```typescript
try {
  const response = await r2Client.send(new GetObjectCommand({ Key: r2Key }));
  const imageBuffer = await streamToBuffer(response.Body);
  archive.append(imageBuffer, { name: outputFilename });
} catch (error) {
  console.error(`Failed to fetch image from R2: ${r2Key}`, error);
  // Skip this image and continue (log error for debugging)
  // OR throw error to fail entire zip generation (stricter approach)
}
```

**Recommendation**: Skip individual image errors but log them. Include note in summary about any skipped images.

**Handle Memory Limits:**
- Use streaming throughout (no buffering)
- `archiver` handles this automatically
- Verify with large batch test (100 images)

**Handle Slow Downloads:**
- Set reasonable timeout on client (5 minutes for 100 images)
- Show progress indicator if possible (browser-dependent)
