## Phase 3: AI Generation Pipeline

### Overview

This phase implements the core AI generation pipeline using RunPod's Seedream 4 Edit model. When a user starts generation, the system converts all uploaded images to PNG, submits jobs to RunPod, polls for completion, downloads results to R2, and provides real-time progress updates.

## Notes

### Kid Model Images (MVP Approach)

For MVP, we're using placeholder stock kid images stored in `public/models/`. This is a temporary solution. Future enhancements include:
- Generating kid models with separate AI call (face generation model)
- Storing generated kid models in R2 for reuse
- Allowing users to upload their own kid model photos

### Polling Frequency

We're using 3-second polling intervals:
- Fast enough for good UX (near real-time updates)
- Slow enough to avoid RunPod API rate limiting
- Can be adjusted if rate limiting issues occur

### Concurrent Job Submission

When submitting large batches (e.g., 100 images):
- Use `p-limit` to limit concurrent RunPod API calls (10 at a time)
- Prevents overwhelming RunPod API
- Reduces chance of rate limiting errors

### Future Migration to Background Workers

For production scalability, consider migrating polling to background workers:
- Use BullMQ or Inngest for job queue management
- Background worker polls RunPod independently of client
- More reliable (continues even if user closes browser)
- Can be added later without changing API contracts

### Error Handling Strategy

All API routes should handle these error scenarios:
- RunPod API unavailable (503 errors)
- Rate limiting (429 errors)
- Image download failures
- R2 upload failures
- Database transaction failures
- Invalid authentication/authorization

Each error should be logged with context for debugging and provide user-friendly error messages in the UI.
