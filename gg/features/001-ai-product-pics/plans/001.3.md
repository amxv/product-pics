# 001-ai-product-pics - Phase 3: AI Generation Pipeline

## Phase Overview

This phase implements the core AI generation pipeline using RunPod's Seedream 4 Edit model. When a user starts generation, the system converts all uploaded images to PNG, submits jobs to RunPod, polls for completion, downloads results to R2, and provides real-time progress updates. After this phase, users can generate product photos with kid models in natural settings.

**Dependencies**: This phase depends on Phase 2 being complete (batches created, images uploaded to R2).

**References**:
- See `gg/features/001-ai-product-pics/summary.md` for the complete feature overview and implementation approach.
- See Phase 1 plan for Database Schema and Shared Type Definitions.
- See Research document for RunPod API details and polling patterns.

#### Patterns, Conventions, and Design Decisions to Reuse

**Generation Flow:**
1. User clicks "Start Generation" on batch details page
2. Client calls `/api/generate/submit` with batchId
3. Server:
   - Validates batch status is "uploaded"
   - Fetches all uploaded images
   - Converts each to PNG (using sharp)
   - Generates random backgrounds (no repeats until all used)
   - Submits jobs to RunPod for each image
   - Creates `generated_image` and `runpod_job` records
   - Updates batch status to "processing"
4. Client starts polling `/api/generate/poll/[batchId]` every 3 seconds
5. Server polls RunPod status for all pending jobs
6. When job completes:
   - Download image from RunPod CDN
   - Upload to R2 at `generated/` key
   - Update `generated_image` status to "completed"
   - Update `runpod_job` status to "completed"
7. If job fails:
   - Increment retry count
   - If retries < 3, resubmit job
   - If retries >= 3, mark as "failed"
8. When all jobs complete (or max retries reached):
   - Update batch status to "completed" or "partial"
   - Set batch `completedAt` timestamp

**Polling Pattern:**
- Client-side polling with `setInterval` (every 3 seconds)
- Stop polling when batch status is "completed", "partial", or "failed"
- Handle browser refresh (resume polling if batch still processing)
- Display real-time progress (completed/total count, progress bar)

**Retry Logic:**
- Exponential backoff: 1s, 2s, 4s between retries
- Max 3 retries per image
- Track retry count in `generated_image` table
- Save error messages for debugging

**Background Diversity:**
- Select backgrounds randomly from BACKGROUNDS array
- No repeats until all backgrounds used (shuffle algorithm)
- Assign background during job submission, store in `runpod_job.background`

#### Key Constraints to work within

- **RunPod polling required** - No webhooks, must poll status endpoint
- **30-minute batch timeout** - Track submission time, timeout if exceeded
- **Max 3 retries per image** - Hard limit, mark as failed after
- **Client-side polling for MVP** - Can migrate to background workers later
- **Image conversion before RunPod** - All images must be PNG
- **RunPod CDN URLs are temporary** - Download to R2 immediately upon completion

**Constants:**
```typescript
const RUNPOD_ENDPOINT = 'https://api.runpod.ai/v2/seedream-v4-edit';

```

**Functions:**

**1. `submitJob(input: RunpodSubmitRequest): Promise<string>`**

Submit job to RunPod:
- Input: Prompt, image URLs, size, safety checker
- Makes POST to `${RUNPOD_ENDPOINT}/run`
- Headers: `Authorization: Bearer ${RUNPOD_API_KEY}`
- Returns job ID from response
- Throws error on failure

**2. `getJobStatus(jobId: string): Promise<RunpodStatusResponse>`**

Check job status:
- Makes GET to `${RUNPOD_ENDPOINT}/status/${jobId}`
- Headers: `Authorization: Bearer ${RUNPOD_API_KEY}`
- Returns full status response (status, output, error)
- Throws error on API failure

**3. `downloadResult(resultUrl: string): Promise<Buffer>`**

Download generated image from RunPod CDN:
- Fetches image from resultUrl
- Returns buffer
- Throws error on failure

**Error Handling:**
- Wrap all fetch calls in try-catch
- Parse error responses from RunPod
- Throw descriptive errors for debugging

### 4. Create Background Selection Utility

**File**: `lib/backgrounds.ts`

**Purpose**: Randomly assign backgrounds with no repeats

**Function:**

**`assignBackgrounds(count: number): Background[]`**

Assign backgrounds to images:
- Input: Number of images in batch
- Logic:
  - Shuffle BACKGROUNDS array
  - If count <= backgrounds.length: return first `count` items (no repeats)
  - If count > backgrounds.length: repeat shuffle and append until `count` reached
- Output: Array of backgrounds (one per image)
- Ensures maximum diversity (no repeats until all backgrounds used)

**Example:**
- 5 images → 5 different backgrounds
- 10 images → all 8 backgrounds + 2 random repeats
- 100 images → 12.5 cycles through all backgrounds

### 5. Create Generation Submit API Route

**File**: `app/api/generate/submit/route.ts`

**Method**: POST

**Purpose**: Start generation for a batch

**Request Body**:
```typescript
{
  batchId: string
}
```

**Logic**:
1. Validate session (must be authenticated)
2. Fetch batch from database
3. Verify user owns batch
4. Verify batch status is "uploaded"
5. Fetch all uploaded images for batch
6. Verify at least 1 image uploaded
7. Assign backgrounds using `assignBackgrounds()`
8. For each uploaded image:
   a. Fetch image from R2 (using r2Client.send(GetObjectCommand))
   b. Convert to PNG using `convertToPNG()` from `lib/image.ts`
   c. Upload PNG to temporary R2 location (`temp/{imageId}.png`)
   d. Generate presigned GET URL for temp PNG (valid 24 hours)
   e. Get kid model image URL (hardcoded or from separate collection - see notes below)
   f. Generate prompt using `generatePrompt()`
   g. Submit job to RunPod using `submitJob()`
   h. Create `generated_image` record with status="processing", background assigned
   i. Create `runpod_job` record with jobId, status="in_queue", prompt, background
9. Update batch status to "processing"
10. Return success

**Response**:
```typescript
{
  success: boolean,
  totalJobs: number
}
```

**Errors**:
- 401 if not authenticated
- 403 if user doesn't own batch
- 400 if batch status is not "uploaded"
- 404 if batch not found

### 6. Create Generation Poll API Route

**File**: `app/api/generate/poll/[batchId]/route.ts`

**Method**: GET

**Purpose**: Poll generation progress for a batch

**Logic**:
1. Validate session
2. Fetch batch from database
3. Verify user owns batch
4. Fetch all `generated_image` records with status="processing"
5. For each processing image:
   a. Fetch latest `runpod_job` for this generated_image
   b. Call `getJobStatus(jobId)` to check RunPod status
   c. Handle status:
      - `IN_QUEUE` or `IN_PROGRESS`: No action, continue
      - `COMPLETED`:
        - Download result using `downloadResult()`
        - Upload to R2 at generated key (`generated/{userId}/{batchId}/{imageId}.png`)
        - Update `generated_image` status="completed", set r2Key, completedAt
        - Update `runpod_job` status="completed", resultUrl, completedAt
        - Increment batch.completedImages
      - `FAILED`:
        - Check retry count
        - If retryCount < 3:
          - Increment retryCount
          - Resubmit job (call `submitJob` again)
          - Create new `runpod_job` record
          - Update `generated_image` with new runpodJobId
        - If retryCount >= 3:
          - Update `generated_image` status="failed", errorMessage
          - Update `runpod_job` status="failed", errorMessage
          - Increment batch.failedImages
6. Check if all images complete:
   - If all images have status="completed": Update batch status="completed", completedAt
   - If some failed: Update batch status="partial", completedAt
   - If all failed: Update batch status="failed", completedAt
7. Check timeout (30 minutes since batch.updatedAt when status became "processing"):
   - If exceeded: Mark remaining images as "failed", update batch status
8. Return current progress

**Response**: `BatchProgress` from `lib/types.ts`
```typescript
{
  batchId: string,
  status: BatchStatus,
  totalImages: number,
  completedImages: number,
  failedImages: number,
  processingImages: number
}
```

**Errors**:
- 401 if not authenticated
- 403 if user doesn't own batch
- 404 if batch not found

**Performance Optimization:**
- Use `Promise.all()` to poll multiple RunPod jobs concurrently (with p-limit to avoid rate limiting)
- Cache job statuses briefly to avoid hammering RunPod API

### 14. Add Timeout Check

**File**: `app/api/generate/poll/[batchId]/route.ts`

**Addition:**

In polling logic, add timeout check:
```typescript
// Check if 30 minutes elapsed since processing started
const processingStartTime = batch.updatedAt; // When status became "processing"
const elapsedMs = Date.now() - processingStartTime.getTime();
const timeoutMs = 30 * 60 * 1000; // 30 minutes

if (elapsedMs > timeoutMs) {
  // Mark all remaining processing images as failed
  await db.update(generatedImageTable)
    .set({
      status: 'failed',
      errorMessage: 'Timeout: Generation exceeded 30 minutes'
    })
    .where(
      and(
        eq(generatedImageTable.batchId, batchId),
        eq(generatedImageTable.status, 'processing')
      )
    );

  // Update batch status
  const failedCount = processingImages.length;
  await db.update(batchTable)
    .set({
      status: completedImages > 0 ? 'partial' : 'failed',
      failedImages: batch.failedImages + failedCount,
      completedAt: new Date()
    })
    .where(eq(batchTable.id, batchId));
}
```

## Notes

### Kid Model Images (MVP Approach)

For MVP, use placeholder stock kid images:
- Find or generate 9 stock images (3 per demographic: baby, boy, girl)
- Store in `public/models/` directory
- Name: `baby-1.jpg`, `baby-2.jpg`, `baby-3.jpg`, `boy-1.jpg`, etc.
- Randomly select one when submitting jobs

**Future Enhancement:**
- Generate kid models with separate AI call (face generation model)
- Store generated kid models in R2 for reuse
- Allow users to upload their own kid model photos

### Polling Frequency

3 seconds is recommended:
- Fast enough for good UX (near real-time)
- Slow enough to avoid rate limiting
- Can be adjusted based on RunPod API rate limits

### Concurrent Job Submission

When submitting 100 jobs:
- Use `p-limit` to limit concurrent API calls (e.g., 10 at a time)
- Prevents overwhelming RunPod API
- Reduces chance of rate limiting

### Migration to Background Workers

For production, consider migrating polling to background workers:
- Use BullMQ or Inngest for job queue
- Background worker polls RunPod every 5 seconds
- Reduces client-side polling overhead
- More reliable (continues even if user closes browser)
- Can be added in future without changing API contracts

### Testing Strategy

Test scenarios:
1. Single image generation (happy path)
2. Batch of 10 images (verify backgrounds are diverse)
3. Failed job (mock RunPod error, verify retry logic)
4. Timeout (mock slow RunPod response, verify 30-min timeout)
5. Browser refresh during processing (verify polling resumes)
6. All images fail (verify batch status="failed")
7. Some images fail (verify batch status="partial")
