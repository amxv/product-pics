# 001-ai-product-pics - Phase 2: Image Upload and Batch Management

## Phase Overview

This phase implements the batch creation workflow and file upload system. Users can create batches with demographic settings, upload product images directly to R2 using presigned URLs, and view their batch history. After this phase, users can manage batches and see uploaded images, but generation is not yet functional.

**Dependencies**: This phase depends on Phase 1 being complete (database, auth, R2 client configured).

**References**:
- See `gg/features/001-ai-product-pics/summary.md` for the complete feature overview and implementation approach.
- See Phase 1 plan for Database Schema and Shared Type Definitions.

#### Patterns, Conventions, and Design Decisions to Reuse

**Presigned URL Upload Pattern:**
1. Client requests presigned URL from API route (POST `/api/upload/presigned-url`)
2. Server validates session, file type, size, and batch ownership
3. Server creates `uploaded_image` record with status="pending"
4. Server generates presigned PUT URL with 15-minute expiry
5. Client uploads file directly to R2 using presigned URL
6. Client notifies server of completion (POST `/api/upload/complete`)
7. Server updates `uploaded_image` status to "uploaded"

**Server Action Pattern:**
- Place in `actions/` directory
- Add `'use server'` directive at top
- Validate session first
- Validate inputs with Zod
- Return typed results with success/error states

**Form Pattern:**
- Use shadcn/ui Form components with react-hook-form
- Validate with Zod schemas
- Display errors inline
- Disable submit during loading
- Show success/error toast notifications

**Image Conversion Pattern:**
- Use `sharp` library server-side to convert all formats to PNG
- Resize to max 2048×2048 (maintain aspect ratio)
- Compress with quality=90
- Store converted PNG in R2

#### Key Constraints to work within

- **Presigned URLs expire in 15 minutes** - Upload must complete within window
- **Max 100 images per batch** - Validate before allowing upload
- **Max 20MB per file** - Validate before generating presigned URL
- **Only authenticated users** - All API routes must validate session
- **User can only access own batches** - Verify ownership in all operations
- **File conversion happens server-side** - Client uploads original, server converts to PNG before R2

### 3. Create Presigned URL API Route

**File**: `app/api/upload/presigned-url/route.ts`

**Method**: POST

**Purpose**: Generate presigned URL for direct R2 upload

**Request Body**: `PresignedUrlRequest` from `lib/types.ts`
```typescript
{
  batchId: string,
  fileName: string,
  fileType: 'image/jpeg' | 'image/png' | 'image/webp' | 'image/heic',
  fileSize: number // bytes
}
```

**Logic**:
1. Validate session (must be authenticated)
2. Validate request body with `presignedUrlRequestSchema`
3. Fetch batch from database, verify:
   - Batch exists
   - User owns batch
   - Batch status is "uploading"
   - Total images < 100 (batch limit)
4. Generate unique image ID with `nanoid()`
5. Create `uploaded_image` record with status="pending"
6. Generate R2 key using helper: `generateUploadKey(userId, batchId, imageId)`
7. Generate presigned PUT URL with 15-minute expiry using `@aws-sdk/s3-request-presigner`
8. Return presigned URL and metadata

**Response**: `PresignedUrlResponse`
```typescript
{
  presignedUrl: string,
  uploadedImageId: string,
  r2Key: string,
  expiresIn: number // 900 seconds (15 minutes)
}
```

**Errors**:
- 401 if not authenticated
- 403 if user doesn't own batch
- 400 if validation fails or batch limit reached
- 404 if batch not found

### 4. Create Upload Complete API Route

**File**: `app/api/upload/complete/route.ts`

**Method**: POST

**Purpose**: Mark upload as complete after client successfully uploads to R2

**Request Body**:
```typescript
{
  uploadedImageId: string,
  success: boolean,
  errorMessage?: string
}
```

**Logic**:
1. Validate session
2. Fetch `uploaded_image` record
3. Verify user owns the batch (via uploaded_image → batch → userId)
4. Update `uploaded_image` status:
   - If success: status="uploaded"
   - If failed: status="failed", save errorMessage
5. Update batch `totalImages` count (increment)
6. Return success

**Response**:
```typescript
{
  success: boolean
}
```

**Errors**:
- 401 if not authenticated
- 403 if user doesn't own batch
- 404 if uploaded_image not found

### 16. Create Image Processing Utility

**File**: `lib/image.ts`

**Functions**:

**1. `convertToPNG(buffer: Buffer): Promise<Buffer>`**

Convert image to PNG format using sharp:
- Input: Image buffer (any format)
- Process:
  - Resize to max 2048×2048 (maintain aspect ratio, only if larger)
  - Convert to PNG
  - Compress with quality=90
- Output: PNG buffer

**2. `validateImageBuffer(buffer: Buffer): Promise<boolean>`**

Validate that buffer is a valid image:
- Use sharp to attempt loading
- Return true if successful, false otherwise
- Catch errors and return false

**Note**: These utilities are used server-side only, not exposed to client.

### 17. Create Upload Helper Utility

**File**: `lib/upload.ts`

**Functions**:

**1. `uploadToR2(presignedUrl: string, file: File, onProgress?: (progress: number) => void): Promise<void>`**

Upload file to R2 using presigned URL:
- Use `XMLHttpRequest` for progress tracking
- Set Content-Type header
- Track upload progress
- Call `onProgress` callback with 0-100 percentage
- Throw error on failure
- Return promise that resolves on success

**2. `retryUpload(fn: () => Promise<void>, maxRetries: number = 3): Promise<void>`**

Retry upload with exponential backoff:
- Attempt upload
- On failure, retry up to `maxRetries` times
- Backoff: 1s, 2s, 4s
- Throw final error if all retries fail

### 19. Add Image Conversion to Upload Flow

**Modification**: `app/api/upload/presigned-url/route.ts`

**Note**: Since presigned URLs allow direct client-to-R2 upload, we cannot convert images server-side during upload. Instead:

**Option 1: Client-side conversion (not ideal, limited browser support for HEIC)**
- Convert to PNG in browser before upload
- May not support all formats (HEIC problematic)

**Option 2: Two-step upload (RECOMMENDED)**
- Client uploads original to R2 at temporary key
- Server processes and converts to PNG asynchronously
- Move to final key
- This happens during generation phase (Phase 3)

**Decision**: Use Option 2. In this phase, upload original files as-is. Conversion happens in Phase 3 before sending to RunPod.
